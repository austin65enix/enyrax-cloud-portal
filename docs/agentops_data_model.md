# AgentOps Data Model

## 1. Purpose

`data/agentops/demo_agent_runs.json` contains the first phase AgentOps normalized agent run records. The demo data is designed for summary APIs, run list APIs, and dashboard development before any real Codex telemetry parser is enabled.

This data model does not save full conversation content. It stores operational metadata only, so AgentOps can measure AI Agent workload, Token cost, tool usage, modified file count, task results, and delivery efficiency without exposing sensitive prompts or responses.

## 2. Privacy Boundary

Do not save:

* full prompt
* full response
* shell output
* file contents
* credentials
* API keys
* `.env` values
* private notes

Only save:

* session metadata
* token usage
* duration
* task name
* project name
* tool call count
* files modified count
* result summary

## 3. Fields

* `id`: Stable normalized run id used by AgentOps.
* `session_id`: Source session identifier from Codex or another agent source.
* `source`: Telemetry source. The first local source is `codex_local`.
* `project_name`: ENYRAX project or module associated with the run, such as `ServiceOps`, `ProjectOps`, `Sync Gateway`, `Vulnerability Inventory`, or `AgentOps`.
* `task_name`: Short human-readable task title.
* `task_number`: Task reference such as `Task #099`.
* `started_at`: ISO 8601 timestamp when the run started.
* `ended_at`: ISO 8601 timestamp when the run ended.
* `duration_seconds`: Run duration in seconds.
* `status`: Normalized run status.
* `result`: Short result summary. This must not contain full prompt, full response, shell output, file contents, or secrets.
* `model`: Agent model reported by the source metadata.
* `input_tokens`: Input tokens counted for the run.
* `cached_tokens`: Context cache reused tokens.
* `output_tokens`: Tokens generated by the AI.
* `reasoning_tokens`: Reasoning / thinking tokens counted for the run.
* `total_tokens`: Total counted Token usage used by dashboard totals.
* `tool_calls`: Count of tool calls such as reads, edits, shell commands, search, and validation.
* `files_modified`: Count of files modified during the run.
* `commands_run`: Count of shell commands run during the session.
* `error_count`: Count of errors observed in normalized metadata.
* `warning_count`: Count of warnings observed in normalized metadata.
* `created_at`: Timestamp when the normalized agent run record was created.

## 4. Status Model

* `success`: The agent completed the requested task or delivered the expected artifact.
* `failed`: The agent did not complete the requested task or validation failed.
* `interrupted`: The session stopped before completion because of cancellation, timeout, restart, or context interruption.
* `unknown`: The parser could not determine the final status.

## 5. Metrics Derived from Demo Data

Future AgentOps APIs and dashboards can derive:

* Agent Runs
* Total Tokens
* Cached Tokens
* Output Tokens
* Reasoning Tokens
* Tool Calls
* Files Modified
* Success Rate
* Avg Duration
* Top Tasks
* Top Projects

## 6. Future Parser

Future parser:

```text
scripts/parse_codex_sessions.py
```

The parser will read metadata from:

```text
~/.codex/sessions/*.jsonl
```

It must follow the AgentOps Privacy Boundary. The parser may extract session metadata, token usage, duration, task hints, project hints, tool call counts, modified file counts, and a short result summary. It must not store full prompt, full response, shell output, file contents, credentials, API keys, `.env` values, or private notes.
