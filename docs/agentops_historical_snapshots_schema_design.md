# AgentOps Historical Snapshots Schema Design

## Overview

AgentOps Historical Snapshots preserve dashboard-level operational metrics so Trend Chart, Release Quality Score, Risk & Anomaly Detection, and review health can be compared across preview generations and releases.

Historical snapshots only store dashboard-level operational metrics.

Historical snapshots do not store:

* prompt / response content
* shell output
* command text
* diff
* file contents
* raw JSONL lines
* credentials
* secrets
* full home paths

Snapshot values are operational indicators, not billing-grade cost data. Project / task values may remain pipeline-level metadata classifications until a future schema split separates pipeline-level and session-level labels.

## Snapshot Types

### 1. Daily Snapshot

Daily snapshots support the Trend Chart and help reviewers observe daily AI Agent / Codex preview metrics.

Daily snapshots may be generated by a scheduled daily job or manually after a preview review.

### 2. Release Snapshot

Release snapshots preserve dashboard state before a release / tag.

Release snapshots track review status, quality score, risk level, token normalization, and classification coverage for a specific version.

### 3. Manual Review Snapshot

Manual review snapshots preserve one reviewer-triggered evidence snapshot after a review command is run.

Manual review snapshots can be used for an audit trail when a scheduled daily or release snapshot is not enough.

* `semantic_warnings` must use allowlisted warning codes and must not include free-text prompt / response excerpts.
* Token values remain operational estimates, not billing-grade cost data.

## Storage Options

### Option A: JSON files under data/agentops/snapshots/

Example:

```text
data/agentops/snapshots/daily/2026-05-31.json
data/agentops/snapshots/releases/v0.6.22-agentops-dashboard-preview.json
```

Pros:

* easy to review
* easy to commit
* suitable for MVP
* no backend dependency

Cons:

* data volume grows over time
* query capability is limited

### Option B: SQLite / PostgreSQL table

Example:

```text
agentops_snapshots
```

Pros:

* queryable
* can power trend APIs
* suitable for a formal dashboard

Cons:

* requires migration / backend API work
* requires stricter schema governance

### Option C: Hybrid

Use JSON files first, then import snapshots into a database when dashboard querying needs grow.

Recommended MVP:

```text
Option A: JSON files under data/agentops/snapshots/
```

Option A keeps the first implementation auditable and avoids backend dependency. A future task can migrate or mirror the same schema into SQLite / PostgreSQL.

## Proposed File Layout

```text
data/agentops/snapshots/
  README.md
  daily/
    2026-05-31.json
  releases/
    v0.6.22-agentops-dashboard-preview.json
```

Layout notes:

* `README.md` records the snapshot safety boundary.
* `daily/` stores daily snapshots.
* `releases/` stores release snapshots.
* Filenames must not contain secrets, full paths, or raw session ids.
* Release tags and dates are acceptable safe filenames.

## Snapshot Generation Inputs

Allowed inputs:

* output of `scripts/review_agentops_preview.py --json`
* safe aggregate metrics from `data/agentops/agent_runs_preview.json`
* release tag
* commit short SHA
* generated timestamp
* dashboard-local quality score rules
* dashboard-local risk indicator rules

Disallowed inputs:

* prompt
* response
* shell output
* command text
* diff
* file contents
* raw JSONL line
* credentials
* full home path
* secret values
* arbitrary source code content
* arbitrary log content

## Snapshot Validation Rules

Future validation should enforce:

* required fields must exist
* numeric fields must be non-negative
* ratios must be between 0 and 1
* `review_status` must be allowlisted: `passed` / `failed` / `warning` / `unknown`
* `snapshot_type` must be allowlisted: `daily` / `release` / `manual_review`
* risk level must be allowlisted: `low` / `medium` / `high` / `critical` / `unknown`
* semantic warning codes must be allowlisted
* no extra fields unless schema version changes
* no forbidden text patterns
* no full home path
* no credentials / secrets
* `preview_file` must be basename only

## Schema Versioning

Each snapshot must include:

```json
{
  "schema_version": "agentops_snapshot_v1"
}
```

Future schema changes should bump `schema_version`.

Schema v1 focuses on aggregate dashboard metrics. Schema v1 does not split `pipeline_project` / `session_project` or `pipeline_task` / `session_task` yet. That schema split should be future work.

## Relationship to Pipeline / Session Schema Split

Historical snapshots v1 can store existing project/task aggregate values, but must preserve a semantic warning that project/task may be pipeline-level classifications.

Future schema split may add:

* `pipeline_project_distribution`
* `session_project_distribution`
* `pipeline_task_distribution`
* `session_task_distribution`

This task does not implement that split.

## Non-goals

* no parser changes
* no backend changes
* no UI changes
* no API changes
* no preview JSON regeneration
* no billing-grade cost reporting
* no content-based inference
* no raw session storage
* no AI answer quality scoring

## AgentOps Snapshot Index Publish Path

Task #133 documents how `/data/agentops/snapshots/index.json` should be served by nginx.

Only `/data/agentops/snapshots/` should be exposed, not the entire `/data/` tree. Snapshot files contain aggregate operational metrics only. They do not contain prompt / response content, command output, raw JSONL, credentials, secrets, or full home paths.

The production nginx site uses `root /var/www/enyrax-portal`, so the publish rule should use `root` consistently. Add a narrow snapshot location and block the remaining `/data/` tree:

```nginx
location ^~ /data/agentops/snapshots/ {
    root /var/www/enyrax-portal;
    default_type application/json;
    add_header Cache-Control "no-store";
    try_files $uri =404;
}

location ^~ /data/ {
    return 404;
}
```

`Cache-Control: no-store` is preferred for preview dashboard freshness. If the snapshot index is unavailable, Trend UI falls back to clearly labeled sample data.
